Implementation · src/pipeline/ner_pipeline.py

PHI detection & masking

  • Regex layer — Not ML. Compiled patterns for structured PHI (e.g. DATE, PHONE, MRN, DOB, AGE, many HOSPITAL name forms).
  • spaCy — Pretrained English pipeline en_core_web_sm (statistical NER). Labels mapped to your schema: PERSON; ORG/FAC → HOSPITAL; GPE/LOC → LOCATION.
  • Spans merged with longest-wins overlap resolution, then replaced (e.g. [PERSON]) per mask_style.
  • API: POST /api/deidentify — same path as the dashboard demo (plus optional file upload elsewhere).
This is a hybrid system: regex for format-driven identifiers, spaCy for context-driven names and places. It is not a single end-to-end transformer model.

Implementation · sklearn · src/pipeline/anomaly_detector.py

Batch anomaly scoring

  • Model: sklearn.ensemble.IsolationForest (n_estimators=100, contamination=0.05, fixed random seed).
  • Preprocessing: sklearn.preprocessing.StandardScaler fit on the submitted batch.
  • Features (9): phi_count, phi_density, text_length, date_count, person_count, phone_count, mrn_count, hospital_count, label_diversity.
  • API behaviour: Each POST /api/anomaly-scan call fits scaler + forest on the notes in the request (minimum 10), then scores those same notes — not a single long-lived production model.

Implementation · src/pipeline/readmission_predictor.py

Readmission-style risk score

  • Not XGBoost. This class does not use gradient boosting. It derives features from each note (PHI counts, density, text length, label diversity, etc.), then compares them to means and standard deviations computed from the training corpus.
  • Score: Weighted sum of z-scores → sigmoid → risk_score in [0,1]; risk_level is low / medium / high by fixed thresholds.
  • First use: If the predictor is not fitted, the API loads up to 2000 rows from processed_notes and requires at least 50 usable notes to fit — otherwise returns 400 with an explicit error.
  • API: POST /api/predict-readmission with a JSON note object or { "notes": [ ... ] } for batch.
For supervised XGBoost on Diabetes-130-style columns, use the fourth tab — POST /api/clinical-risk-model/predict — only when a pickle is loaded (see status endpoint).

Implementation · src/models/clinical_risk_model.py

Tabular readmission (XGBoost)

  • Model: xgboost.XGBClassifier via ClinicalRiskModel; artifact path at startup: CLINICAL_RISK_MODEL_PATH or data/models/clinical_risk_model.pkl.
  • Training data: CSV like data/raw/diabetic_data.csv; target readmitted with positive class <30.
  • Features: src/models/feature_engineer.py — not interchangeable with the NLP note JSON in tab 3.
  • API: GET /api/clinical-risk-model/status (loaded + eval snapshot), POST /api/clinical-risk-model/predict with { "records": [ { …row… }, … ] }.
Demo only: scores are not validated for clinical decisions. If status shows loaded: false, train and save a pickle or point CLINICAL_RISK_MODEL_PATH at an existing file and restart the app.
